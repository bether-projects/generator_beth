Token Features
What's the difference between 100k, Fixed, Capped and Unlimited Supply?

100k
The number of tokens available to you will be 100.000. During deployment, the whole token supply will be produced and given to the Token Owner wallet. You can't change the supply later.

Fixed Supply
During deployment, the whole token supply will be produced and given to the Token Owner wallet. You can't change the supply later.

Capped Supply
You can specify an initial supply that will be sent to the Token Owner's wallet. You can later increase or decrease supply by minting or destroying tokens (if permitted). You will not be able to create more tokens than the supply cap.

Unlimited Supply
You can specify an initial supply that will be sent to the Token Owner's wallet. You can later increase or decrease supply by minting or destroying tokens (if permitted). You will be able to generate an endless number of tokens with no upper restriction.

What's the difference between None, Ownable and Role Based Access?

None
Because there aren't any operations that require rights, your token doesn't need an access type.

Ownable
Your Token will be assigned an Owner. By default, the account used to deploy your Token will be the owner and will be allowed to mint new tokens or execute the finish minting method. Token ownership can be transferred to addresses or Smart Contracts.

Role Based
Roles will be assigned to your Token. Accounts having the role "MINTER" will be able to create new tokens. Accounts having the "ADMIN" role can add or remove roles from minters and other admins. By default, the accounts used to deploy your Token will be ADMIN and MINTER. Furthermore, your Token will have the Ownable behavior.

Who can mint or burn tokens?

Mint
Depending on the Token Access Type you have. Only the Token Owner will be allowed to create new tokens if you select Ownable Access. Only addresses with the MINTER role will be able to create new tokens if you select role-based access. If you select a Capped supply in either scenario, you won't be able to mint more tokens than the specified cap. Instead, selecting Limitless supply will allow you to create an unlimited number of tokens.

Burn
Everyone will be allowed to burn the tokens he was holding. Only with consent can a third party burn tokens from other addresses. Nobody will be allowed to burn tokens from other addresses without permission, not even the Token Owner.

What is Operable Token?

There is no mechanism to execute code after a BET20 transfer or approval (i.e. making a payment), so to perform an activity, another transaction must be sent and GAS must be paid twice. Operable Token simplifies token payments and works without the need for any other listener. It enables a callback following a transfer or approval in a single transaction.
There are numerous proposed applications for Betherance Smart Contracts that accept BET20 payments.
Examples could be:

  • to create a token payable crowdsale
  • selling services for tokens
  • paying invoices
  • making subscriptions

What is Token Recover?

Many tokens have been lost forever in Smart Contracts. Every Betherance contract has the potential to be a token trap for BET20 tokens. They cannot be recovered, resulting in financial damages for end users.
TokenRecover enables the contract owner to recover any BET20 token that was sent into the contract by mistake. BET20 Token Recover is responsible for this behavior.

Betherance Blockchain and Ecosystem
What is a DApp?

DApp is an acronym for decentralized application.
The backend code of a DApp runs on a decentralized peer-to-peer network. In contrast, consider an app where the backend code is hosted on centralized servers.
A DApp, like any other app, can have frontend code and user interfaces written in any language that can make calls to its backend. Its frontend can also be hosted on decentralized storage systems like Swarm or IPFS.
To be deemed a Dapp (pronounced Dee-app, similar to Email), an application must meet the following criteria:

  • The application must be totally open-source, independent, and without a single party controlling the bulk of its tokens. The application's protocol may be modified in response to proposed enhancements and market feedback, but all modifications must be approved by a majority of its users.
  • To avoid any central points of failure, the application's data and operational logs must be cryptographically saved in a public, decentralized blockchain.
  • The program must employ a cryptographic token (bitcoin or a token unique to its system) for access, and any value contribution from (miners / farmers) should be paid in the application's tokens.
  • The program must issue tokens in accordance with a standard cryptographic algorithm that serves as proof of the value nodes contribute to the application (Bitcoin employs the Proof of Work Algorithm).

What is a BET20 token?

BET20 is a Betherance token standard that extends BET20, the most prevalent Betherance token standard. Consider it a template for tokens that describes how they can be used, who can spend them, and other usage constraints. It is interoperable with Betherance's BET20 due to their similarities.
BET20 was designed as a technical standard for Betherance, with the purpose of providing a versatile structure for developers to create a variety of tokens. These could be anything from firm shares to dollars deposited in a bank vault (i.e., a stablecoin).

What is GAS and how to set Gas price?

The term "Gas" refers to a particular unit used in BSC. It quantifies the amount of "effort" required to complete an action or group of actions.
Every operation that can be carried out by a transaction or contract on the BSC platform has a set gas cost, with more computationally intensive operations having higher gas costs than operations with fewer computationally intensive operations.
Gas is crucial since it makes sure the right cost is being paid by transactions sent to the network, which is why. We prevent the network from becoming clogged with completing a lot of intensive work that isn't helpful to anyone by mandating that a transaction pay for each operation it performs (or causes a contract to perform).
